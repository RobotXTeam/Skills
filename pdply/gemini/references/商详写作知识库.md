# 商详写作知识库

本文档用于指导 agent 根据 `/home/steven/agent/PDPly/参考数据` 中的 PRD、datasheet、规格表、图片说明、链接资料等，生成一份结构完整、内容完整、格式正确、精炼不流水账的商品详情文档，并最终输出为 docx。

写作目标不是机械照抄《商详标准》，而是参考标准中的必填项和后台结构，同时学习历史商详的实际写法：重点产品写完整、有场景、有说服力；普通配件或低重要性产品写清楚必要信息即可，少写营销长文。

## 1. 总原则

1. 先判断产品类型和重要性，再决定写多长。
2. 用户最关心的信息优先：这是什么、解决什么问题、核心卖点、关键规格、适配/兼容对象、包装包含什么、在哪里看文档。
3. 不为了填满标准而写。标准是上架字段参考，不是每个产品都要写满。
4. 不编造参数、认证、兼容性、SKU、链接、库存、价格、折扣、上市时间。
5. 缺信息允许留空，但必须用清晰占位，方便人工补齐。例如：`[待补：产品图片包]`、`[待补：Datasheet 链接]`、`[待确认：兼容型号]`。
6. 内容要像人写的商详：短句为主，有判断、有取舍，不堆砌 datasheet。
7. 英文商详正文默认用英文；中文说明只用于内部提示、占位或特殊备注。
8. 输出 docx 时，保留清晰标题层级、表格和列表，不需要复杂样式。

## 2. 产品分级与篇幅决策

生成前必须先给产品分级。分级决定模块数量、文字长度、表格颗粒度。

### A 级：重点标品 / 平台型产品 / 高客单价产品

适用对象：

- 开发板、边缘 AI 设备、机器人、网关、完整套件、核心模组、新产品线主推产品。
- 用户购买前需要理解能力边界、应用场景、生态、文档和配件。
- 参考样本：Jetson Developer Kit Bundle、reBot Arm Bundle、Wio-S3、Home Assistant Starter Kit、MeshCore 套件。

建议篇幅：

- Short description：60-120 words。
- Features：5-8 条，每条 1-2 句。
- Specification：1-3 个表格，核心规格完整。
- Application：3-5 个场景，每个 1-2 句。
- Document/Learn/SEO/Part List 必写。
- 可写 FAQ、Bundle Option、Also add、Related SKU，但只在资料支持时写。

### B 级：普通标品 / 功能模块 / 专用硬件

适用对象：

- 无线模块、摄像头配件、传感器、通信模块、显示板、开发配件。
- 有明确技术规格和适配场景，但不需要长篇品牌故事。
- 参考样本：RTL8852BE WiFi Module、T1000-E Adapter、M12 Lens、Wio-LR2021。

建议篇幅：

- Short description：40-80 words。
- Features：4-6 条。
- Specification：1 个主表，必要时拆成 Mechanical / Electrical / Wireless。
- Application：1-4 个场景，能写清楚即可。
- Hardware overview：有接口、pinout、尺寸、兼容对象时写；否则可留空。
- Document、Part List、SEO 必写。

### C 级：低重要性配件 / 线材 / 替换件 / 简单耗材

适用对象：

- FPC cable、lens frame、servo 单个替换件、转接头、外壳、小配件。
- 用户只需要知道“用于什么、适配谁、规格是多少、盒内有什么”。
- 参考样本：FPC Cable、Small Lens Frame、Servo Motor。

建议篇幅：

- Short description：25-60 words。
- Features：0-4 条。没有真实卖点时不要硬写。
- Specification：一个 2 列表格，5-10 行。
- Application：1 句或 1 条链接即可。
- Hardware overview：有尺寸图、pinout 或安装关系才写。
- Learn、You May Like This、Also add 可空。
- 不写长篇故事，不写泛泛场景。

## 3. 生成前的信息抽取

从 `参考数据` 中读取资料后，先整理以下信息。不要直接开始写正文。

### 必抽信息

- SKU：如果没有，写 `[待补：SKU]`。
- Product name / proposed title。
- 产品类型：标品、模块、套件、配件、线材、镜头、开发板、整机等。
- 核心芯片/平台/主控/传感器/接口。
- 核心卖点：最多 8 个，按购买决策重要性排序。
- 关键规格：尺寸、电源、接口、无线协议、性能指标、工作温度、兼容系统等。
- 兼容对象：适配哪些主产品、型号、版本。
- 包装清单：Part List。
- 文档链接：datasheet、user manual、wiki、getting started、3D file、schematic、GitHub。
- 认证/法规信息：只在资料明确提供时记录。
- 图片素材：主图、场景图、尺寸图、pinout、hardware overview。

### 需要判断的信息

- 用户买它是为了解决什么问题？
- 哪些参数真正影响购买？
- 哪些信息只适合放 Document，而不适合放正文？
- 是否有同型号不同版本，需要 Related attribute SKUs？
- 是否是套件，需要 Bundle Option？
- 是否有必需配件，需要 Also add？
- 是否有同类产品可比较，需要做选型表？

## 4. 推荐输出结构

docx 中按以下顺序输出。标题文字可以保留中英混合，方便内部上架。

```text
产品图片【Images And Videos】
产品名称【Product Name】
产品简述【Short description】
关联SKU【Related attribute skus】
商品详情【Content】
  5.1 产品特点【Features】
  5.2 产品规格【Specification】
  5.3 产品硬件图【Hardware overview】
  5.4 产品应用【Application】
  5.5 产品文档【Document】
Attribute
产品零件列表【Part List】
产品wiki/blog【Learn】
猜你喜欢【You May Like This】
关联产品推荐【Also add】
SEO【Search Engine Optimization】
产品特殊通知【Product notes】
产品广告位/活动【Product Events】
产品 Bundle 页面【Product bundle】
产品 Selection 页面【Product selection】
```

可省略规则：

- C 级配件可以保留标题但内容留空，或只输出必要模块。推荐保留后台字段标题，方便人工补齐。
- `Certification / ECCN / HTS` 通常不写正文，除非资料明确要求；可以写一句内部提示：`[由品质/认证平台维护，若缺失请人工确认]`。
- `Product Events`、`Product bundle`、`Product selection` 没有资料时留空。
- `FAQ` 不是标准必填项，只给复杂套件、生态型产品或用户疑问明显的产品写。

## 5. 各模块写法

### 5.1 产品图片【Images And Videos】

写图片素材状态，不要生成图片本身。

必须包含：

- `[待补：产品图片包]` 或图片文件名。
- 如果资料有图，列出建议用途：主图、白底图、场景图、Part List 图、尺寸图、pinout、hardware overview。

建议写法：

```text
[待补：产品图片包]
Suggested image order:
1. Main product image
2. Feature or real-use scenario image
3. Part list image
4. Dimension image
5. Pinout / hardware overview image
```

规则：

- 标准要求至少 4 张白底图、最多 6 张商品图，最后一张建议为尺寸图。
- 重要产品需要真实场景图；简单配件可用主图、尺寸图、适配关系图即可。
- 不引用外部图片作为商详主图。

#### 图片语义判断

写商详时必须把图片当成内容证据，而不是装饰。参考资料里的图片要先抽取、看缩略图、判断用途，再决定是否插入。

常见图片类型和位置：

- 产品渲染图、实物图、爆炸图：放在 `产品图片` 或 Features 开头，帮助用户建立产品形态认知。
- 板卡堆叠、结构拆解、外壳、接口布局、pinout、尺寸、安装孔：放在 `Hardware overview`。这些图的意义是降低集成风险，让用户知道怎么接、怎么装、怎么散热。
- Web UI、SenseCraft、模型转换、API、Dashboard 截图：放在 Features 或 Application。意义是证明“开发流程简单”和“可部署”，不是为了展示界面本身。
- 场景图、检测结果、行业应用 demo：放在 `Application`。意义是让用户看到产品如何进入真实场景。
- 表格截图：优先转成 Word 表格；只有视觉比较、界面截图或无法可靠转录时才保留为图片。
- 竞品截图、市场调研截图、内部 PRD 备注图：默认不放客户商详，除非用户明确要求写竞品对比。

插图数量：

- A 级产品：建议 4-8 张有明确目的的图。
- B 级产品：建议 1-4 张。
- C 级产品：0-2 张，通常只需要主图、尺寸图或适配关系图。

插图要求：

- 不要硬上图片。参考资料里没有合适图片，就写占位并放入缺失清单，让用户或设计补图。
- 不能为了“有图”而随便找一张相似图、低价值图、内部截图或不匹配图放进去。
- 正式商品图、Part List 图、场景图、尺寸图缺失时，必须明确写 `[待补：正式商品图]`、`[待补：Part List 图]` 等，不得用 PRD 截图顶替。
- 每张插图前后要有一句短说明或 caption，解释为什么放在这里。
- 不要把低清、内部敏感、纯市场调研、无客户价值的截图塞进商详。
- 图片应服务购买决策：形态、性能、连接、安装、软件流程、应用效果。

### 5.1.1 DOCX 格式参考

在 PDPly 工作区生成 docx 时，默认参考：

`/home/steven/agent/PDPly/配件或套件文档/MeshCore虚拟套件上架商祥.docx`

格式要点：

- 不使用 Word Heading 样式做大标题，使用普通段落。
- 一级模块用 `1. 产品图片【Images And Videos】` 这种编号标题。
- 编号为蓝色，标题文字加粗。
- Content 子模块如 `5.2 产品规格【Specification】（必选）` 使用普通加粗段落。
- 正文使用 Arial，约 10.5pt，段落间距紧凑。
- 图片宽度默认约 5.75 inch。
- 表格使用普通网格表，不做复杂样式。

### 5.2 产品名称【Product Name】

产品名称要简洁，但要包含核心关键词和关键型号。

公式：

```text
[Product type / positioning] + [core model or chipset] + [1-2 key capabilities] + [compatibility or application if important]
```

好标题特征：

- 用户一眼知道是什么。
- 包含主芯片、接口、协议、适配对象中最重要的 1-2 个。
- 不堆满所有参数。
- 不把 SEO 全塞进产品名，SEO 放 Meta Title 和 Description。

示例方向：

- `Wio-S3 Wireless Module with IPEX, ESP32-S3R8 and SX1262, LoRaWAN, Wi-Fi & BLE`
- `FPC Cable - for Raspberry Pi Camera 3 Wide`
- `SenseCAP Tracker T1000-E Magnetic Charging Data Adapter, 7.62mm 4-Pin Pogo Pin to USB-C`

### 5.3 产品简述【Short description】

Short description 是首屏核心，不写流水账。

结构：

1. 第一句：It is / This is + 产品定位 + 适配对象/核心平台。
2. 第二句：核心卖点 + 关键参数。
3. 第三句：适合的应用或购买理由。
4. 如果是配件，补一句兼容对象。

长度：

- A 级：60-120 words。
- B 级：40-80 words。
- C 级：25-60 words。

不要做：

- 不重复完整产品名。
- 不写超过 1 段。
- 不塞太多营销形容词。
- 不出现未确认的 “best / most powerful / industry-leading”。

模板：

```text
It is a [product type] designed for [target device / scenario]. Built around [core chip/platform/interface], it provides [key capability 1] and [key capability 2]. With [important spec/compatibility], it is suitable for [main application], while keeping integration simple for [target user].
```

### 5.4 关联SKU【Related attribute skus】

只在有同类型不同型号、频段、容量、接口、颜色、版本时写。

格式：

```text
{"SKU1":"Version name","SKU2":"Version name","SKU3":"Version name"}
```

规则：

- 必须包含本 SKU。
- 不确定 SKU 时写 `[待补：关联 SKU]`，不要编。
- 不把 Also add 配件混进 Related attribute skus。
- 同类产品推荐放 You May Like This，必需配件放 Also add。

### 5.5 商品详情【Content】

Content 的核心顺序：

1. Features
2. Specification
3. Hardware overview
4. Application
5. Document

这 5 个 tab 中，Features 和 Specification 通常必写；其他按资料和产品类型决定。

#### 产品特点【Features】

功能：

- 帮用户快速理解为什么买。
- 给设计、广告、第三方平台直接提炼卖点。

格式：

```markdown
- **Feature Name:** One concise sentence explaining the value, with concrete parameters when available.
```

数量：

- A 级：5-8 条。
- B 级：4-6 条。
- C 级：0-4 条。没有真实卖点时可留空或只写 2-3 条。

排序：

1. 核心性能或核心能力。
2. 兼容性 / 生态 / 协议。
3. 易用性 / 预装 / 即插即用。
4. 可靠性 / 工业级 / 温度 / 认证。
5. 尺寸 / 低功耗 / 结构设计。

写法要求：

- 每条 feature 名称 2-6 个词，使用粗体。
- 描述里尽量写具体参数：TOPS、GHz、dBm、mm、V、protocol、interface、OS、temperature。
- 不要把规格表逐行改写成 feature。
- 不要写 “High Quality”、“Easy to Use” 这类空话，除非后面有具体支撑。

好例子：

```markdown
- **Long-Range Wireless Connectivity:** Supports LoRaWAN on EU868 and US915 bands, while also providing 2.4GHz Wi-Fi and BLE 5.0 for flexible IoT deployment.
- **Compact Magnetic Charging:** The 7.62mm 4-pin pogo pin connector aligns magnetically with the T1000-E tracker for stable charging and data transfer.
```

#### 产品规格【Specification】

规格统一用表格，优先 2 列，最多 4 列。

2 列表格适合大多数产品：

```markdown
| Specification | Details |
|---|---|
| Main Chip | [待补] |
| Interface | [待补] |
| Power Supply | [待补] |
| Dimensions | [待补] |
| Operating Temperature | [待补] |
```

拆表规则：

- A 级复杂产品可拆成 `System`、`Mechanical`、`Wireless`、`Power`、`I/O`、`Environment`。
- B 级通常 1 个表即可。
- C 级一个 5-10 行的表即可。

表格尺度：

- 单个表格建议 5-18 行。
- 超过 20 行时考虑拆表或把完整 datasheet 放 Document。
- 不要把所有 datasheet 内容都搬进商详。
- 一格里多项内容可用 `<br>` 分行，但不要超过 4 行。

必须注意：

- 型号、接口、尺寸、电源、兼容对象、频段、温度范围必须准确。
- 如果规格缺失，写 `[待补：xxx]`。
- 不把未经确认的认证写进规格。

#### 产品硬件图【Hardware overview】

适合写：

- pinout
- dimension
- connector mapping
- interface layout
- installation relation
- button / LED / port overview

不适合写：

- 大段功能介绍。
- 多张重复图片。
- 与规格表完全重复的文字。

数量：

- A 级：1-4 张图说明。
- B 级：1-3 张。
- C 级：有图才写，没图留空。

写法：

```text
[待补：Pinout image]
Pin correspondence:
| Pin | Function |
|---|---|
| 1 | [待补] |
```

#### 产品应用【Application】

Application 写用户场景，不写泛泛行业名。

数量：

- A 级：3-5 个。
- B 级：2-4 个。
- C 级：1 个即可。

格式：

```markdown
- **Smart Agriculture:** Enables remote environmental monitoring and long-range data collection for irrigation, soil, and weather sensing nodes.
```

规则：

- 每个场景 1-2 句。
- 有搭配产品时可自然提及，但不要变成广告列表。
- 简单配件可以写成一句：`Designed as a replacement/accessory for [compatible product].`
- 如果需要配图，建议用规整表格，不要一张大图配一句话占满页面。

#### 产品文档【Document】

Document 用于承接完整资料，避免正文过长。

表格格式：

```markdown
| Name | Link |
|---|---|
| Datasheet | [待补：公开 PDF 链接] |
| User Manual | [待补：公开链接] |
| Wiki / Getting Started | [待补：公开链接] |
| 3D File | [待补：公开链接] |
| GitHub | [待补：公开链接] |
```

规则：

- 链接必须是客户可访问的公开链接。
- 本地文件、飞书云盘、企业内部链接不能作为最终客户链接。
- 如果目前只有本地 datasheet，写 `[待补：上传公开 PDF 链接]`。

### 5.6 Attribute

Attribute 是后台分类信息，不需要文学化。

常见字段：

```text
Product Type:
Categories:
Series:
RAM:
Storage:
OS:
Network:
Sensors:
```

规则：

- 从现有资料或历史同类产品推断时要谨慎。
- 不确定就写 `N/A` 或 `[待补]`。

### 5.7 产品零件列表【Part List】

Part List 非常重要，涉及客诉、售前咨询、退换货。

格式：

```markdown
| Item | Quantity |
|---|---|
| [Product / accessory name] | x1 |
```

规则：

- 必须准确。
- 不确定就写 `[待确认：Part List]`。
- 套件必须列出所有包含物。
- 如果有选配件，不要写进 Part List，应放 Bundle Option 或 Also add。

### 5.8 产品wiki/blog【Learn】

用途：

- 放 Wiki、Blog、Project、教程、案例。
- 优先级：Wiki > Blog > Project > Others。

格式：

```markdown
| Type | Title | Description | Link |
|---|---|---|---|
| Wiki | Getting Started with [Product] | Setup guide and basic usage. | [待补] |
```

规则：

- 没有资料可留空。
- 不编链接。
- 同产品线的通用教程可以放，但标题要准确。

### 5.9 猜你喜欢【You May Like This】

用途：

- 推荐同品类、同系列、同应用方向的 SKU。

规则：

- 可以列出候选，但没有 SKU 或后台配置资料时留 `[待补]`。
- 不和 Also add 混淆。
- You May Like This 是相似或替代产品；Also add 是搭配购买的配件。

### 5.10 关联产品推荐【Also add】

用途：

- 推荐让主产品跑起来的必需品，或主场景常用配件。

排序：

1. 刚需配件：电源、线缆、天线、存储、散热、安装件。
2. 场景增强：摄像头、传感器、扩展板、外设。
3. 可选升级：更大 SSD、更强天线、外壳等。

规则：

- 建议不超过 5 个。
- 不放同类替代产品。
- 不确定库存和仓位时只写候选，不写强结论。

### 5.11 SEO【Search Engine Optimization】

SEO 需要写，但不牺牲可读性。

字段：

```text
Meta Title:
Meta Description:
Tags:
```

Meta Title：

- 主关键词放开头。
- 末尾可加 `- Seeed Studio`，除非标题已经过长。
- 尽量 60 characters 左右。
- 不堆参数。

Meta Description：

- 1-2 句完整英文。
- 包含主关键词和 1-2 个长尾关键词。
- 尽量 150-170 characters；复杂产品可略长，但不要变成段落。
- 写用户价值，不堆关键词。

Tags：

- 5-15 个。
- 包含产品线、芯片、接口、应用、品牌词。
- 不确定的 tag 不写。

示例：

```text
Meta Title: Wio-S3 Wireless Module with ESP32-S3 and SX1262 for LoRaWAN IoT - Seeed Studio
Meta Description: Wio-S3 integrates ESP32-S3, SX1262 LoRa, Wi-Fi, and BLE for long-range IoT and edge AI applications, with IPEX and non-IPEX hardware options.
Tags: Wio-S3, ESP32-S3, SX1262, LoRaWAN, Wi-Fi, BLE, IoT module, edge AI
```

### 5.12 产品特殊通知【Product notes】

只写购买前必须知道的信息。

适合内容：

- 需要焊接。
- 只兼容特定版本。
- 不包含电源/电池/线材。
- 需要额外网关或软件。
- 大货/批量购买联系邮箱。

规则：

- 最多 1-3 条。
- 不把普通说明放 Note。
- 语气客观，不吓人。

### 5.13 Bundle / Selection

只有在资料明确提供时写。

Bundle Option 适合：

- 套件销售。
- 必选主件 + 可选配件。
- 多个电源线地区版本。

格式：

```markdown
#### [Group Name]（必选，单选）
- [SKU] Product Name

#### [Accessory Group]（可选，多选）
- [SKU] Product Name
```

Selection 适合：

- 同类 SKU 很多，用户需要按参数选型。
- 用问题引导选择，例如频段、接口、容量、颜色、用途。

## 6. 不同产品类型的写法

### 开发板 / 模组

重点：

- 主控/芯片
- 无线协议
- 内存/Flash
- 接口
- 电源
- SDK / firmware / platform
- 典型应用
- 认证

Feature 顺序建议：

1. Processing / Wireless capability
2. Memory / performance
3. Connectivity / protocol
4. Hardware flexibility
5. Ease of development
6. Reliability / certification

### 边缘 AI / Jetson / 工控设备

重点：

- 算力、CPU/GPU、内存、存储
- JetPack / software stack
- I/O 和扩展
- 电源和散热
- 应用场景：robotics、vision AI、industrial automation
- 文档和开发资源

正文可以比普通产品长，但仍然避免复制平台介绍。

### 套件 / Bundle

重点：

- 套件解决什么完整问题。
- 每个组件在系统里负责什么。
- 是否即插即用 / 预配置。
- Bundle Option 和 Part List 必须清楚。
- FAQ 很有价值，建议写 3-6 个。

写法：

- Features 可用系统角色组织，例如 Brain / Sensor / Bridge。
- Specification 不必列每个组件完整 datasheet，只列主件核心规格，并把完整资料放 Document。

### 配件 / 线材 / 转接头

重点：

- 适配对象
- 接口类型
- 长度/尺寸/pin 数
- 功能：充电、数据、连接、固定、替换
- Part List

不要写：

- 大段行业应用。
- 虚假的“高性能”。
- 与主产品无关的背景。

### 镜头 / 结构件

重点：

- 适配设备
- 焦距、FOV、分辨率、安装规格、尺寸
- 使用效果或应用距离
- 是否手动调焦
- Part List

### 舵机 / 电机 / 运动部件

重点：

- 电压
- 转速 / 扭矩 / 齿比
- 协议
- 尺寸 / 重量
- 预配置 ID 或适配位置
- 注意事项：负载、过热、行程限制

## 7. 写作风格

### 应该像这样

- 具体：`Supports EU868 and US915 LoRaWAN frequency plans.`
- 有场景：`Designed for low-power sensor nodes in smart agriculture and industrial monitoring.`
- 有取舍：`For full electrical details, refer to the datasheet in the Document section.`
- 简洁：每段 2-4 句，每条 feature 1-2 句。

### 不应该像这样

- 空泛：`This product is high quality and widely used.`
- 堆词：`AI IoT smart industrial powerful advanced reliable solution.`
- 流水账：把 datasheet 每一行改写成自然语言。
- 过度营销：`the best choice for all users`。
- 未确认：`FCC/CE certified`，但资料没有明确证明。

## 8. 缺失信息处理规则

允许缺信息，但要可补。

占位格式：

- `[待补：SKU]`
- `[待补：产品图片包]`
- `[待补：公开 Datasheet 链接]`
- `[待确认：Part List]`
- `[待确认：是否兼容 Raspberry Pi 5]`
- `[待确认：认证信息]`

不要：

- 不要写“无”来掩盖缺失。
- 不要凭经验补参数。
- 不要使用内部路径作为客户链接。

## 9. docx 格式要求

生成 docx 时建议使用以下样式：

- 文档标题：Heading 1。
- 一级模块：Heading 2。
- Content 下子模块：Heading 3。
- Bundle 分组或 FAQ 问题：Heading 4 或加粗。
- Features/Application：项目符号列表。
- Specification/Part List/Document/Learn：表格。
- 表格第一行加粗。
- 占位符保留为普通文本或加粗均可。

表格建议：

- Specification：2 列为主。
- Document：2 列或 4 列。
- Learn：4 列。
- Part List：2 列。
- 不做复杂合并单元格，避免后续编辑困难。

文件命名：

```text
[SKU] [Product Name] 商详.docx
```

如果 SKU 缺失：

```text
[Product Name] 商详.docx
```

## 10. 生成流程

agent 生成商详时按以下步骤执行：

1. 读取 `参考数据` 中所有可用文件，优先提取 PRD、datasheet、规格表、图片说明、已有链接。
2. 输出内部草稿前，先判断产品等级：A / B / C。
3. 整理信息抽取表，标出缺失和待确认项。
4. 根据等级选择篇幅，不套用全量标准。
5. 先写 Product Name、Short description、Features，这三项决定商详方向。
6. 再写 Specification、Hardware overview、Application。
7. 最后写 Document、Part List、Learn、SEO、Also add 等后台字段。
8. 精简一轮：删除重复、空泛、无依据、过长内容。
9. 检查所有参数、SKU、链接、兼容性是否来自资料。
10. 输出 docx，同时可附带一个简短的缺失信息清单。

## 12. 最终质量检查清单

生成完成前逐项检查：

- [ ] 是否判断了产品等级，并按等级控制篇幅？
- [ ] Product Name 是否包含产品类型和关键型号？
- [ ] Short description 是否 1 段内说清楚产品、卖点和用途？
- [ ] Features 是否按重要性排序，且有具体参数支撑？
- [ ] Specification 是否用表格，且没有搬运过量 datasheet？
- [ ] Application 是否是真实场景，而不是泛泛行业词？
- [ ] Hardware overview 是否只放接口、尺寸、pinout、安装等直观信息？
- [ ] Document 是否只使用公开客户可访问链接；没有链接则占位？
- [ ] Part List 是否准确；不确定是否明确待确认？
- [ ] Related SKU、You May Like This、Also add 是否没有混淆？
- [ ] SEO 是否自然，不堆关键词？
- [ ] 是否删除了无意义营销词和流水账？
- [ ] 是否没有编造认证、兼容性、SKU、参数、链接？
- [ ] docx 标题层级、表格、列表是否清楚？

## 13. 给后续 Skill 的使用说明

后续可将本知识库封装为一个 skill。skill 的职责应是：

- 输入：一个参考数据目录，例如 `/home/steven/agent/PDPly/参考数据`。
- 读取：本知识库、参考数据中的 docx/pdf/md/xlsx/txt/图片文件名。
- 输出：一份 docx 商详文档。
- 附加输出：缺失信息清单和人工确认项。

skill 执行要求：

1. 先读取本知识库，不要直接套《商详标准》。
2. 从参考数据抽取事实，事实不足时留占位。
3. 自动判断 A/B/C 级。
4. 默认英文正文，中文标题保留。
5. 默认生成完整后台结构，但 C 级产品允许内容极简。
6. 输出 docx，不只输出 Markdown。
7. 若用户要求“精简”，优先删长段描述、FAQ、复杂应用，不删 Product Name、Short description、Features、Specification、Part List、SEO。
8. 若用户要求“完整”，也不要堆资料；完整指字段完整、事实完整、购买决策信息完整。

推荐 skill 内部产物：

```text
output/
  [SKU or Product Name] 商详.docx
  [SKU or Product Name] 缺失信息清单.md
```

缺失信息清单模板：

```markdown

- [待补：公开 Datasheet 链接]
- [待确认：Part List]

- [待确认：Related SKUs]
```
