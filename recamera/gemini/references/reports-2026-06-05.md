## Wiki QA Report: UDP Face Analysis with reCamera

Source:
- Wiki file: `Applications/UDP_Face_Analysis.md`
- Public URL: `https://wiki.seeedstudio.com/recamera_udp_face_analysis/`
- Tested on: 2026-06-05, `seeed` at `192.168.4.7` / `192.168.42.197`, reCamera `192.168.42.1`, OS `0.2.3`

Result:
- Direct wiki run: fail
- Final run after fixes: pass
- Score: 6/10

Evidence:
- Official receiver video: `/home/steven/下载/recamera_udp_face_wiki_visual.mp4` (35 s, 640x700).
- Key logs: UDP FPS around 18 when no face attributes are triggered; YOLO inference around 43-45 ms. The rerun with `single 0.5 1` recorded a green face box plus red `face0` age/gender/race/emotion text in the official receiver window.

Direct-run findings:
- `multi` mode failed with `ModelFactory::create failed`; local face model required `single`.
- Running without `LD_LIBRARY_PATH` failed with missing CVI/media libraries.
- Running without `sudo` failed in CVI runtime device memory initialization.

Fixes applied:
- Stopped Node-RED/SSCMA services.
- Added `LD_LIBRARY_PATH=/mnt/system/lib:/mnt/system/usr/lib:/mnt/system/usr/lib/3rd`.
- Ran with `sudo`.
- Used `single 0.7 1` instead of `multi 0.5 3`.
- Used the repository-provided `udp_receiver.py` under Xvfb and recorded the OpenCV window with `ffmpeg x11grab`.

What the wiki should change:
- Add exact runtime `sudo env LD_LIBRARY_PATH=...` command.
- Explain when to use `single` vs `multi` and match the supplied model.
- Mention Linux/headless validation options such as Xvfb screen recording, and clarify that visual success depends on a clear face in the camera view.

## Wiki QA Report: YOLO11n Benchmarking on reCamera

Source:
- Wiki file: `Applications/yolo_benchmark.md`
- Public URL: `https://wiki.seeedstudio.com/recamera_benchmarking/`
- Tested on: 2026-06-05, `seeed` at `192.168.4.7` / `192.168.42.197`, reCamera `192.168.42.1`, OS `0.2.3`

Result:
- Direct wiki run: partial/fail on Linux because receiver details are Windows-only and runtime env is omitted.
- Final run after fixes: pass
- Score: 7/10

Evidence:
- Official receiver video: `/home/steven/下载/recamera_yolo_benchmark_wiki_visual.mp4` (24 s, 640x700).
- Key logs: repeated `total` times around `49-58ms`; receiver logs showed 1-4 targets per frame and the recorded video shows red boxes/ID labels.

Direct-run findings:
- Google Drive folder was usable and contained `recamera_benchmark`, detection model, segmentation model, and `yolo_udp.py`.
- Wiki does not name exact files in text.
- Runtime requires `sudo` and `LD_LIBRARY_PATH`.
- UDP port is fixed at `5001`; command only passes receiver IP.
- Provided receiver uses `cv2.imshow`, which is inconvenient for headless Linux validation.

Fixes applied:
- Downloaded assets with `gdown --folder`.
- Ran with `sudo env LD_LIBRARY_PATH=...`.
- Used the Drive-provided `yolo_udp.py` under Xvfb and recorded the OpenCV window with `ffmpeg x11grab`.
- After reCamera reboot, re-read the current seeed USB interface IP before starting the sender. It changed from `192.168.42.197` to `192.168.42.20` and then `192.168.42.55`.

What the wiki should change:
- Name all Drive files explicitly.
- Add Linux/macOS receiver instructions and a headless screen-recording option.
- Document fixed UDP port `5001`.
- Add required `sudo env LD_LIBRARY_PATH=...` command.
